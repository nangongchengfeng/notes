## 核心问题：为什么需要结构化Prompt？
写Prompt的本质目标是让大模型的**输入 → 思考 → 输出**效果更稳定、更可控、更专业。

大模型内部推理逻辑是线性的：
```
理解角色 → 理解背景 → 理解任务 → 推理 → 按格式输出
```
**结构越清晰 → 推理越稳定**，结构化Prompt比随意的一句话Prompt效果提升50%以上。

## 工程级标准Prompt结构（7要素）
业界最稳定的标准结构：
```
Role + Context + Task + Constraints + Thinking + Output Format + Examples
```
对应中文：
```
角色 → 背景 → 任务 → 约束 → 思考方式 → 输出格式 → 示例
```

### 完整模板（可直接套用）
```text
# Role
你是一个专业的 {角色}，擅长 {核心能力清单}

# Context
当前背景：
{业务背景、环境信息、已知条件}

# Task
你的任务是：
{具体要完成的目标}

# Constraints
必须遵守：
1. {约束1，比如：不胡编乱造，命令必须可执行}
2. {约束2，比如：只给技术方案，不输出无关内容}
3. {约束3，比如：敏感操作必须提示人工确认}

# Thinking
请按照以下步骤思考：
1. 分析问题
2. 判断原因
3. 给出方案
4. 评估风险
5. 输出结果

# Output Format
请按照如下格式输出：
### 问题分析
### 解决方案
### 风险提示
### 最终结果

# Example
输入：{示例输入}
输出：{示例输出}
```

## 三种主流工程常用Prompt框架
### 1️⃣ RCTO 框架（最推荐，适合AI运维场景）
| 字段 | 含义 |
|------|------|
| Role | 角色 |
| Context | 背景 |
| Task | 任务 |
| Output | 输出格式 |

#### 示例（K8s故障排查）
```text
你是一个Kubernetes专家，精通云原生运维、故障排查

当前背景：
集群CPU使用率持续升高，Pod频繁重启，版本为K8s 1.27

你的任务：
分析可能原因并给出排查步骤

输出格式：
1. 可能原因（按概率排序）
2. 排查命令（可直接执行）
3. 解决方案
```

**适用场景**：AI运维、自动化分析、智能体、日志分析、告警分析

---

### 2️⃣ CO-STAR 框架（国际通用，OpenAI推荐）
| 字段 | 含义 |
|------|------|
| C (Context) | 背景 |
| O (Objective) | 目标 |
| S (Style) | 输出风格 |
| T (Tone) | 语气 |
| A (Audience) | 受众 |
| R (Response) | 输出要求 |

#### 示例（运维报告生成）
```text
Context:
你正在分析Kubernetes集群上周的异常告警数据，共产生127条告警，其中80%是CPU使用率过高

Objective:
生成运维分析周报

Style:
正式技术报告
Tone:
专业严谨
Audience:
SRE团队工程师
Response:
Markdown格式，包含问题统计、根因分析、优化建议三部分
```

**适用场景**：报告生成、总结分析、技术文档、AI助手

---

### 3️⃣ ReAct 框架（Agent专用，目前主流AI Agent标配）
标准结构：
```
Thought → Action → Observation → Final Answer
```
#### 示例（自动化故障排查）
```text
Thought: 用户需要分析服务器CPU过高的问题，首先需要查看系统进程占用情况
Action: 执行top -bn1命令查看CPU占用前10的进程
Observation: 输出显示java进程占用CPU 98%，PID 12345
Thought: 接下来需要查看java进程的堆栈信息，判断是不是死循环或者GC问题
Action: 执行jstack 12345 > jstack.log命令导出堆栈
Observation: 堆栈显示有大量的死循环线程在执行xx业务逻辑
Final Answer: 问题根因为xx业务代码死循环，建议立即回滚最近上线的xx版本
```

**适用场景**：AI智能体、自动化运维、OpenClaw、LangChain、Claude Code

## AI运维专属标准Prompt模板（可直接复用）
针对运维开发、AIOps场景定制，所有告警分析、日志排查、K8s问题分析都可以统一用这套模板：
```text
# Role
你是一个资深SRE运维专家，精通Linux、Kubernetes、Docker、Prometheus、日志分析、故障排查，有10年以上生产环境运维经验

# Context
当前系统出现异常：
{告警信息、错误日志、监控截图}

系统环境：
{操作系统版本、K8s版本、中间件版本、架构信息}

# Task
请分析问题原因并给出可落地的解决方案

# Thinking
请按照以下步骤分析：
1. 判断告警类型和级别
2. 分析所有可能的原因（按概率从高到低排序）
3. 给出可直接执行的排查命令
4. 提供分阶段的解决方案
5. 评估操作风险，给出规避方案

# Constraints
必须遵守：
1. 不胡编乱造，所有命令必须在Linux/K8s环境中可直接执行
2. 只输出技术相关内容，不要无关的客套话
3. 高危操作（比如rm -rf、重启核心进程、修改生产配置）必须标注【高危操作】，提示人工确认
4. 如果信息不足无法判断，明确说明需要补充什么信息

# Output Format
### 告警分析
### 可能原因（按概率排序）
### 排查命令（可直接执行）
### 解决方案（分临时止损和永久修复）
### 风险提示
```

**使用效果**：
✔ 模型不会胡说八道，输出信息准确
✔ 思考流程可控，不会跳步
✔ 输出结构稳定，可直接对接自动化系统
✔ 可以直接作为Agent的系统Prompt使用

## 进阶：AI Agent四层Prompt架构
真正的生产级Agent系统会用四层Prompt结构，实现完全可控：
```
┌─────────────────┐
│ System Prompt   │ 控制模型整体行为、角色、约束
├─────────────────┤
│ User Prompt     │ 用户输入的具体问题
├─────────────────┤
│ Tool Prompt     │ 控制工具调用的规则、参数、权限
├─────────────────┤
│ Output Schema   │ 严格控制输出结构，方便后续程序解析
└─────────────────┘
```
这是目前所有企业级AI Agent的标准架构。

## 落地建议
### 优先级选择
1. **首选**：`Role + Context + Task + Thinking + Output` 结构，工程稳定性⭐⭐⭐⭐⭐，适合90%的运维场景
2. **报告生成类**：用CO-STAR框架，效果⭐⭐⭐⭐
3. **Agent/自动化类**：用ReAct框架，效果⭐⭐⭐⭐⭐

### 最佳实践
1. 统一团队所有Prompt的模板，不要每个人写自己的风格，输出会更稳定
2. 把常用场景的Prompt做成模板库，直接复用，不需要每次重写
3. 持续优化Prompt效果，把每次模型输出不好的case沉淀下来，更新模板
