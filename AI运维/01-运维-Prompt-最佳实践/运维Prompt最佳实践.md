## **Prompt 1: Linux服务器巡检脚本生成**          



```bash
# Role
  你是拥有10年以上金融行业运维经验的Linux运维架构师，精通Bash脚本企业级开发规范，熟悉等保2.0安全要求，擅长编写可落地、高可靠、无安全隐患的运维自动化脚本，输出的代码可直接纳入生产环境代码库。                                                             
  
  # Context
  当前企业线上服务器数量超过1000台，运维团队之前使用的巡检脚本逻辑不统一、错误处理缺失、存在安全漏洞，经常出现漏检、误报问题，需要统一标准化服务器巡检脚本：
  1. 脚本需要支持对接现有自动化运维平台，可被定时任务、Ansible、Jenkins等上层系统调度
  2. 输出结果需要满足合规审计要求，日志可追溯3个月以上
  3. 所有操作必须符合企业安全规范，不能引入命令注入、敏感信息泄露等风险

  # Task
  编写符合企业级生产规范的Linux服务器巡检Bash脚本，必须严格满足以下要求：
  ## 核心功能
  1. 巡检覆盖6大类维度：
     - 系统基础信息：CPU使用率、内存使用率、磁盘使用率、1/5/15分钟系统负载、uptime运行时间、内核版本
     - 磁盘健康：所有分区inode使用率、只读分区检测、挂载点是否正常挂载
     - 网络状态：关键业务端口监听状态检查、DNS解析测试、公网连通性检测、网卡入/出流量统计
     - 服务状态：sshd、crond、rsyslog、firewalld/iptables等默认关键服务运行状态，支持自定义扩展服务列表
     - 安全检查：最近24小时SSH登录失败次数统计、root用户远程登录权限检查、/etc/passwd系统用户异常变化检测
     - 日志检测：扫描/var/log/messages、/var/log/syslog近1小时内的ERROR级别的关键告警
  ## 硬性规范
  1. 错误处理要求：
     - 脚本开头强制添加`set -euo pipefail`严格运行模式
     - 所有系统命令执行都要有错误捕获逻辑，非预期错误立即输出ERROR级别告警
     - 自定义错误码体系：0=执行成功，1=参数错误，2=权限不足，3=巡检项异常，4=依赖缺失
     - 异常退出时自动清理所有临时文件，避免残留垃圾文件
  2. 日志输出要求：
     - 统一日志格式：`[YYYY-MM-DD HH:MM:SS] [级别] 内容`，级别分为INFO/WARN/ERROR三级
     - 日志同时输出到控制台和`/var/log/server_inspect_${日期}.log`文件
     - 支持日志大小限制，单日志文件超过100M自动轮转，最多保留7天日志
  3. 安全规范要求：
     - 绝对禁止硬编码任何敏感信息（密码、密钥、内网地址等）
     - 所有用户传入参数、动态变量都要做合法性校验，避免命令注入风险
     - 日志输出自动脱敏，隐藏IP、密钥等敏感信息
     - 脚本本身建议权限700，生成的临时文件权限强制为600
  4. 代码规范要求：
     - 函数式编程，每个巡检项封装为独立函数，职责单一，可单独调用
     - 变量名见名知意，全局常量大写，局部变量使用local声明
     - 缩进统一4空格，代码结构清晰无冗余
     - 所有可配置项集中在脚本开头，支持用户自定义巡检阈值（如磁盘使用率告警阈值90%、SSH登录失败告警阈值10次）
  5. 注释要求：
     - 脚本头部必须包含：功能说明、版本号、作者、创建日期、使用示例、依赖检查清单
     - 每个函数头部必须有功能注释、参数说明、返回值说明
     - 关键逻辑步骤加单行注释说明
     - 复杂判断逻辑加注释说明判断依据

  # Thinking
  你需要按照以下逻辑思考输出内容：
  1. 先设计脚本整体架构：开头配置区 → 工具函数区（日志、错误处理、公共方法） → 巡检函数区 → 主执行逻辑 → 退出清理逻辑
  2. 思考每个巡检项的实现方式，优先使用系统原生命令，减少第三方依赖
  3. 思考错误处理覆盖点：确保每个命令执行失败都能被捕获，不会出现脚本异常退出无日志的情况
  4. 思考安全校验逻辑：所有动态输入、变量拼接命令的地方都要加校验，避免命令注入
  5. 思考日志脱敏逻辑：对输出的IP、端口等敏感信息做替换处理
  6. 编写完代码后自行检查是否符合所有要求，有没有遗漏的规范点

  # Output
  输出完整可直接运行的Bash脚本，不需要额外解释，脚本末尾附加使用说明和配置示例。
```



##  Prompt 2: MySQL备份校验脚本生成   



```bash
  # Role                                                                                                                                                                                                                        
  你是拥有8年以上金融行业DBA经验的资深MySQL运维专家，精通数据库备份容灾方案设计，熟悉等保2.0数据安全合规要求，擅长编写零故障、可审计、符合企业生产规范的Bash运维脚本，输出的代码可直接纳入核心数据库运维体系落地。

  当前企业线上MySQL实例超过200个，此前备份体系存在严重风险：1）多套备份脚本逻辑不统一，部分脚本无校验逻辑，曾出现备份文件损坏无法还原的事故；2）部分脚本硬编码数据库密码，存在敏感信息泄露风险；3）无统一错误处理，备份失败无法及时感知。
  现在需要标准化MySQL备份脚本，满足以下背景约束：
  1. 脚本需对接现有自动化运维平台，支持定时任务调度、多实例批量备份
  2. 备份全流程可审计，日志需保留6个月以上满足合规要求
  3. 严格禁止任何可能导致数据库性能抖动、数据损坏的操作
  4. 备份文件需满足数据脱敏要求，不能泄露业务敏感信息

  # Task
  编写符合企业级生产规范的MySQL备份+自动校验Bash脚本，必须严格满足以下所有要求：
  ---
  ## 核心功能
  ### 1. 备份能力
  - 支持两种备份模式：mysqldump逻辑全量备份、xtrabackup物理增量备份
  - 备份文件自动压缩（支持gzip/zstd两种算法可选），命名格式强制为：`mysql_backup_{模式}_{实例名}_{时间戳}.{后缀}`
  - 支持多实例备份，可通过配置文件批量配置多个MySQL实例连接信息
  - 备份过程自动处理锁表逻辑，避免备份数据不一致，锁表时间控制在秒级
  ### 2. 校验能力
  - 备份完成后自动生成SHA256校验和，与备份文件同目录存储，命名格式：`{备份文件名}.sha256`
  - 支持可选开启自动还原测试校验：自动将备份文件临时还原到测试实例，验证数据完整性和可用性
  - 所有备份记录关联存储：包含备份大小、耗时、校验状态、校验人、还原测试结果等元信息
  ### 3. 生命周期管理
  - 支持自定义备份保留天数，自动清理超过保留期的旧备份文件和对应的校验记录、元信息
  - 备份执行前自动检查目标磁盘剩余空间，剩余空间不足备份文件预估大小的1.5倍时提前告警，不执行备份
  ---
  ## 硬性规范
  ### 1. 错误处理
  - 脚本开头强制添加`set -euo pipefail`严格运行模式
  - 每个执行步骤都有错误捕获逻辑，备份/校验任意步骤失败立即终止流程，输出ERROR级日志
  - 备份失败时自动清理不完整的临时备份文件、临时目录，避免占用磁盘空间
  - 自定义错误码体系：0=执行成功，1=参数错误，2=磁盘空间不足，3=备份执行失败，4=校验失败，5=配置文件非法
  ### 2. 日志输出
  - 统一日志格式：`[YYYY-MM-DD HH:MM:SS] [日志级别] [实例名] 日志内容`，日志级别分为INFO/WARN/ERROR三级
  - 日志同时输出到控制台和`/var/log/mysql_backup_${instance_name}_${日期}.log`文件
  - 关键节点必须打日志：备份启动、备份完成、校验启动、校验完成、旧备份清理完成、执行失败
  ### 3. 安全规范
  - MySQL连接信息必须从外部配置文件读取，绝对禁止硬编码数据库密码、密钥等敏感信息
  - 脚本启动时自动检查配置文件权限，配置文件权限必须为600，不符合要求直接终止执行
  - 生成的备份文件权限强制设置为600，仅所有者可读
  - 所有日志输出自动脱敏，隐藏数据库密码、敏感业务表名、用户密码等信息
  ### 4. 代码规范
  - 所有可配置项集中在脚本开头，支持用户自定义：备份存储路径、保留天数、压缩算法、是否开启还原校验、测试实例连接信息等
  - 功能完全模块化：备份函数、校验函数、清理函数、告警函数、上传函数独立封装，可单独调用
  - 变量命名清晰见名知意，避免全局变量污染，所有局部变量使用local声明
  - 支持命令行参数传入：`--instance`指定备份实例、`--mode`指定备份模式、`--config`指定配置文件路径、`--silent`开启静默模式
  ### 5. 注释要求
  - 脚本头部必须包含：功能说明、版本号、使用示例、依赖清单、配置文件格式示例
  - 每个函数头部必须有功能注释、参数说明、返回值说明
  - 备份、校验核心步骤加单行注释说明逻辑
  - 异常处理逻辑加注释说明错误场景和对应处理方式
  ---
  ## 额外扩展功能
  1. 预留备份上传钩子，备份校验成功后可自动调用钩子上传到对象存储/S3/远程服务器
  2. 预留告警钩子，备份/校验失败时可自动调用钩子发送企业微信/邮件/短信告警
  3. 执行完成后输出备份统计报表：备份文件大小、备份耗时、校验结果、磁盘剩余空间
  4. 支持静默运行模式，仅输出ERROR级别日志，适合定时任务无人值守调用

  # Thinking
  你需要按照以下逻辑输出脚本：
  1. 先设计整体架构：配置区 → 命令行参数解析 → 公共工具函数（日志、错误处理、脱敏） → 核心功能函数（备份、校验、清理、告警、上传） → 主执行逻辑 → 异常退出清理逻辑
  2. 优先使用MySQL官方原生工具，减少第三方依赖，兼容MySQL 5.7/8.0所有主流版本
  3. 所有与数据库交互的命令都要加超时限制，避免长时间锁表影响业务
  4. 所有动态变量、用户传入参数都要做合法性校验，避免命令注入风险
  5. 备份锁表逻辑要适配不同业务场景：非核心业务允许短时间锁表，核心业务使用无锁备份模式
  6. 写完代码后自行核对所有要求是否都覆盖，有没有遗漏的规范点

  # Output
  输出完整可直接运行的Bash脚本，不需要额外解释说明，脚本末尾附加配置文件示例和使用说明。
```

5. 所有方案要考虑风险：比如重启服务会不会导致数据丢失，流量切换会不会有雪崩风险，高危操作必须标注风险等级
